## 問題回答演練

### 1. 如何向沒有區塊鏈經驗的工程師介紹區塊鏈運作?

#### Web 3 應用的本質

本質上 Web 3 應用的願景就是將所有 Web 2 的應用運行在一個抽象的 claude service 上．無法被企業或政府單方面的強製關停．也無法在背地暗箱操作修改規則，所有動作公開透明．

#### 把撮合引擎一步步去中心化

我會用一個我一直想做但還沒時間完成的架構實驗來講，題目是：怎麼把一個 OrderBook Matching-Engine，一步一步做成去中心化架構?
我會邊做邊解釋這一步，我們拿掉了哪一個原本依賴的信任．

#### 起點（純 Web2）

一般 orderbook 是 single instance、single thread，嚴格照順訂單序撮合、產生 trade_match、丟給結算模組。為了高可用，先設計做成一個 50 台機器的 cluster：只有一台 master 撮合並產生 trade_match，完成後把這批訂單廣播給其他 49 台 slave。訂單入口設計成 batch —— 這就對應到 block 的概念：用跟 Kafka 一樣的緩衝窗口，order 累積到 threshold 或 ticker 觸發，就把 queue 裡的 order 打包成一個 block。

master 照序撮完，把整個 block 廣播給 49 台 slave 用一模一樣的順序重跑。block 內嚴格照序、block 之間也照序，只要 input 跟順序一樣，每台算出來的狀態就一樣——這就是分布式狀態機。

現在鋪好了去中心化地基：資料複製到 50 台、每台都能自己重算，這讓 **驗證 master 有沒有作弊** 變得可能。不過 master 仍是被信任的中心，順序還是它說了算。

#### 第一步去中心化：拿掉 master

第一件事是拿掉 master —— 誰出下一個 block 不能再固定，要用共識公平地選出下一個出塊者。假設我們有一套成熟共識能隨機公平挑出產塊者，block 間隔固定 12 秒，給塊足夠時間廣播執行。且塊與塊之間使用 hash 串連防止跳塊．或破壞順序．

這一步拿掉的信任:**信任有一個固定可信的 master 來定序** ，以及**信任歷史不被偷改**（hash 串鏈）。

> 補充：如果是「隨機選」，我為什麼不能開一萬個假節點來灌票、讓自己一直被選中?所以共識一定要配 Sybil resistance——PoW 用算力成本、PoS 用質押成本，讓「製造身份」變得很貴。另外 block 之間我會用前一塊的 hash 串起來，而不只是邏輯照序;這樣有人想偷改某一塊歷史，後面所有 hash 全部對不上，立刻被抓。


#### 先誠實面對一個代價

這個實驗我一直想不到怎麼解決得問題是 matching-engine 的強一致性需求。

完全去中心化的非同步網路，有機率 A 出完塊、B 還沒收到就被選成下一個出塊者，造成分岔。撮合引擎對一致性非常敏感，這其實是去中心化撮合的第一個根本矛盾——沒辦法保證強一致。對應 CAP，區塊鏈技術選了可用性跟分區容錯，換來**最終一致**而不是**強一致**

去中心化代價：一致性從「強一致」降級成「最終一致」，這是整個取捨的核心。

#### 第二步：防止節點偽造交易

如果 50 台機器是 50 個不同的人在管，誰都有機會竄改交易、傳假 block。怎麼防?

使用者下單時，用自己的私鑰對訂單 payload 簽名——這裡要用 ECDSA，再丟進 OrderPool。

假設惡意出塊節點竄改了某張單（比如改金額），其他驗證節點收到後，先用 ecrecover 從 (payload, r/s/v) 還原地址。
這是我自己實作驗過的: payload 一被改，ecrecover 不會報錯，而是還原出一個完全不相干的地址。所以光看簽名跟 payload，節點其實判斷不出**合不合法**，因為它總是能還原出某個地址，只是不是交易發起方本人的。

解法是給每個 account 一個 nonce，隨該地址交易數遞增，下單時把當前 nonce 帶進 payload。這樣驗證節點還原出錯誤地址後，一比對 nonce 對不上，就知道這筆被動過手腳，直接拒收整個 block (nonce 同時也擋掉重放攻擊)。

這一步拿掉的信任: **信任出塊節點不會偽造或竄改使用者的交易**（簽名 + ecrecover + nonce，讓任何節點都無法替你下單或改你的單）。


#### 第三步：防止節點偷改帳本

現在能防交易偽造了，但還有一個漏洞:怎麼防節點偷改結算結果，也就是類似直接改 DB 的餘額?

把整個 account 系統做成一棵 Merkle 樹。每個帳戶是一個葉子，葉子兩兩 hash 往上算，最後收斂成一個 state root。任何一個帳戶餘額被動到，那個葉子 hash 變，一路傳到 root 也跟著變。然後每個廣播出去的 block 都附上撮合結算後的 state root。所有驗證節點同步完、自己重算一次 root 跟 block 裡的核對:對得上代表帳本完全一致;對不上代表自己或出塊者其中一方的結算被動過手腳。

這一步拿掉的信任: **信任節點不會偷改已經存好的帳本狀態**（state root commitment，讓全網結算可被每個人獨立驗證）。

參考過去自己寫的文章：https://ithelp.ithome.com.tw/articles/10345090

補上幾個重點：
* 防雙花問題：nonce ↔ 花一次就作廢(最重要):文章已經寫了 UTXO 這半「檢查是否尚未被花費」, account 模型沒有 UTXO 可以作廢,所以改用 nonce 來防重放/雙花。

### 3. 請挑選一個你最熟悉的區塊鏈應用，說明它為什麼適合用區塊鏈技術，而不是傳統資料庫？

我覺得當一個應用的核心價值就是 "無需信任任" ,而且 User 願意承受鏈上成本還有 bug 不可逆風險的代價時,才值得上鏈;否則傳統 DB 更好。」這反而顯得你不是無腦吹區塊鏈。順帶一提,敢講「其實大部分東西不該上鏈」會加分。

我個人最常用的區塊鏈 dApp 是 AAVE。

* 無託管 / 去信任保管(最關鍵): 你的錢是被一份公開、誰都能驗證、沒人能偷改的合約鎖著,不是被某家公司保管。傳統 DB 一定有個 admin 能改任何一行、能凍結或挪用你的錢。這就是 why-not-DB 的核心,而且你 Q1 建立的「DB 有可信 admin、區塊鏈把它拿掉」那條主線在這裡直接套用。殺手級對照:2022 的 Celsius、FTX 就是「用 DB 做的 AAVE」——有個可信中心保管大家的錢,然後跑路 / 作假;AAVE 沒有任何 admin 的私鑰能單方面把錢拿走。

* 可組合性(money legos): AAVE 的存款部位(aToken)能被同鏈上其他協議直接讀取、原子性地拿去當抵押或收益來源。這種跨協議「插積木」只有在開放、共享、可程式化的帳本上做得到;各家公司的 DB 鎖在自己 API 後面,沒辦法原子組合。

* 可驗證的償付能力:準備金、利率、清算即時公開, 都是可以自己驗證的。DB 裡的數字是私有的或者清算價有可能會被偽造且無法驗證, 你只能信對方的單方面說法．


不過去中心化是光譜,不是開關。AAVE 用了可升級的 proxy,owner / governance 能改合約邏輯;為了制衡才有治理代幣,但代幣一旦集中在少數鯨魚手上,又回到寡頭。再加上 oracle 價格來源也是中心化風險點。所以我會說 AAVE 這類智能合約應用是相對去中心化。
